# External AI Review Protocol

This protocol defines the rules and formatting for utilizing external AI models (e.g., Claude, Gemini, Perplexity, Grok, and similar tools) to review content, architecture, and documentation within the HUB_Optimus repository. 

External AI models are valuable tools for analysis, code review, and feedback. However, their outputs must remain strictly advisory. AI outputs must never bypass GitHub issue and Pull Request (PR) governance.

This protocol explicitly aligns with the AI access matrix defined in [Voxterrae/HUB_Optimus#1584](https://github.com/Voxterrae/HUB_Optimus/issues/1584).

## Principles

1. **Advisory Only:** External AI models remain strictly advisory and never become the source of truth. 
2. **GitHub as Source of Truth:** No external finding can become implementation work unless represented by a GitHub issue or PR. Direct action based on external AI output without governance tracking is strictly prohibited.
3. **No Integration:** This protocol governs manual text exchange. Automated integration with external AI vendors is out of scope.
4. **Data Security:** Never upload private secrets, credentials, or non-public repository data to external tools.

## Output Handling Rules

All findings generated by external AI models must be triaged back into GitHub:

- **Findings and Suggestions:** If an external AI review generates useful findings, a human contributor or authorized internal agent must copy the relevant suggestions into a GitHub issue or PR comment.
- **Disagreements:** If the AI model highlights a disagreement or conflict, it must be evaluated by a human or authorized agent. If deemed valid, it should be resolved via standard consensus processes within a GitHub PR or issue.
- **Follow-up Work:** If the AI suggests additional work, an explicit GitHub issue must be created to track it.
- **Direct Action Prohibited:** The output of an external AI cannot be piped directly into runtime, roadmap, or governance contracts without manual formatting and GitHub-centric tracking.

## Review Packet Format

To ensure external models are given the appropriate boundaries and context, all review requests must use the standardized Review Packet format below. This packet can be shared externally without granting authority.

### Standard Review Packet Template

```markdown
### 1. Context
[Provide the high-level goal of the review. For example: "Review this pull request for documentation consistency and alignment with HUB_Optimus governance principles."]

### 2. Files and Scope
[List the specific files, code snippets, or documentation sections under review.]
- File 1: `path/to/file.md`
- File 2: `path/to/code.py`

### 3. Questions
[Specify what the AI should focus on. Be explicit to prevent hallucination or scope creep.]
- Does the documentation explicitly align with the Layer 0 Kernel principles?
- Are there any logical inconsistencies in the proposed scenario?
- Does the code adhere to the required security checks?

### 4. Constraints
[Provide boundaries for the AI model.]
- You are acting in an advisory role only. You have no authority to approve or merge these changes.
- Do not propose sweeping architectural rewrites.
- Focus strictly on the files provided in the scope.
- Maintain the integrity-first perspective.

### 5. Expected Output
[Define the format you expect the AI to return.]
- A bulleted list of specific findings.
- For each finding, provide the file name and the suggested change.
- A brief rationale based on systemic evaluation, not personal preference.
```
